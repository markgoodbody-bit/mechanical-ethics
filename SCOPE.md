# Scope and Claim Ceiling

This repository is a public working surface for the **Correction Window Test v0.1**.

It is deliberately narrow.

## Current public object

The current public object is a practical diagnostic:

> If a system is wrong, can the affected person get correction before the consequence hardens?

The test asks whether a claimed safeguard can still change the affected person's path before the consequence becomes difficult, costly, or impossible to reverse.

## What this repository may claim

This repository may claim:

- the Correction Window Test is a public v0.1 diagnostic;
- it helps inspect whether safeguards can act in time;
- it can be applied to AI systems, automated decisions, public services, institutions, and software systems;
- it can return live protection, partial protection, after-the-fact review, false safeguard, or unknown as working labels;
- it is intended to be used, criticised, narrowed, extended, or broken.

## What this repository may not claim

This repository may not claim:

- Mechanical Ethics is validated;
- the test is reliability-tested as an instrument;
- the test proves a system is ethical;
- the test proves a system is safe, lawful, fair, legitimate, unbiased, or aligned;
- AI alignment is solved;
- timing explains every form of harm;
- public-record examples prove the wider framework;
- a large repository is evidence of correctness.

## Build rule

A file earns a place in this repository only if a stranger could use it without the wider theory and it sharpens the diagnostic rather than enlarging the claim.

Everything else belongs outside the public front door.

## Entry rule for concepts

A concept may enter the public repo only if it passes all four tests:

1. It can be stated in plain language.
2. It supports a concrete diagnostic question.
3. It does not require belief in Mechanical Ethics / TRACE.
4. It does not raise the claim ceiling.

If it fails any of those tests, keep it out, translate it, or place it in archive with a warning that it is not public-front-door material.

## What stays out of the front door

Keep out of the front door:

- glyph systems and internal notation;
- private project mythology or lineage;
- whole-book theory claims;
- untested instrument claims;
- validation language;
- AI-alignment-solved language;
- large taxonomies that do not improve use of the test;
- examples without clear source, caveat, or uncertainty discipline.

## Archive rule

Archive is allowed, but archive is not authority.

Archived material should not control the public front door.

A reader should be able to use the Correction Window Test without reading archived theory, old drafts, book material, or internal project context.

## Size rule

Repo growth is a cost.

More files can improve usability, but they can also dilute the front door, create maintenance debt, and make the work look more proven than it is.

Prefer:

- replace over accumulate;
- examples over theory;
- short files over long files;
- clear caveats over confident reach;
- one usable route over many unfinished routes.

## Stop condition

After adding a batch of files, re-read the repository as a stranger.

If the repo is harder to understand in sixty seconds, the batch failed even if every individual file is good.
