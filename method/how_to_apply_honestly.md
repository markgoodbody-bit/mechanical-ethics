# How to Apply the Correction Window Test Honestly

This note is about discipline.

The Correction Window Test is useful only if it prevents false reassurance and false certainty.

## 1. Name the affected person's clock

Do not ask only how long the institution takes.

Ask how long the affected person has before the relevant consequence hardens.

Examples:

- rent due;
- job filled;
- medical window closes;
- payment stops;
- deadline passes;
- account access loss causes secondary harm;
- label copies downstream;
- evidence becomes unavailable.

A system clock and a person clock are not the same.

## 2. Do not infer safety from missing evidence

If the record does not show timing, authority, evidence access, or correction outcomes, the result is Unknown.

Unknown is not safety.

Unknown is not failure.

Unknown means the public record does not support the stronger claim.

## 3. Separate formal route from live route

A formal appeal, complaint, audit, human review, or dashboard may exist.

That does not answer the test.

Ask whether the route can still alter the affected person's path before hardening.

## 4. Identify the carrier

A safeguard needs a carrier.

A carrier may be pause power, reversal power, deadline, evidence access, affected-person trigger, rollback, compensation authority, external escalation, or enforcement.

If there is no carrier, the safeguard may be only a label.

## 5. Do not force a crisp label

Use a crisp working label only where the evidence supports it.

If the evidence is mixed, say mixed.

If the evidence is missing, say Unknown.

A forced classification is worse than an honest uncertainty.

## 6. Separate case result from system result

A result for one case path does not prove the whole system.

A hiring appeal may be after-the-fact for rejected applicants but live for internal candidates.

A benefit challenge may be partial for one claimant type and false for another.

A medical prioritisation route may be live for urgent symptoms and too late for slower deterioration.

Name the route you are testing.

## 7. Separate diagnosis from design

The test first diagnoses whether correction can act in time.

Design suggestions come after diagnosis.

Do not let an attractive repair idea make the current safeguard look stronger than it is.

## 8. Do not treat size as evidence

A longer document, larger checklist, or bigger repository is not proof.

The question is whether a stranger can apply the diagnostic more clearly.

If a file adds theory but not usable judgement, cut it or archive it.

## 9. Preserve bad results

If the test returns Unknown, False Safeguard, or After-the-Fact Review, keep that result visible.

Do not smooth it into more comfortable language.

The value of the test depends on its ability to say no.

## 10. Use safe result wording

Preferred form:

```text
This safeguard appears to be [working label] because [reason], and the key uncertainty is [uncertainty].
```

Examples:

```text
This safeguard appears to be after-the-fact review because the appeal can correct the record but usually cannot restore the original job opportunity after the role is filled.
```

```text
This safeguard is unknown because the public record does not show whether the affected person can trigger review before the label is copied downstream.
```

```text
This safeguard appears to be live protection because a credible challenge pauses the action, the reviewer has reversal authority, and the deadline fits inside the hardening window.
```

## One-line discipline

Name the person, name the clock, name the carrier, and do not turn missing evidence into reassurance.
