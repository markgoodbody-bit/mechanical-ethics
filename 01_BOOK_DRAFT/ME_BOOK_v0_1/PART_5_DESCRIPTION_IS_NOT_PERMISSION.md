# Part 5 — Description Is Not Permission

Status: first draft. Incomplete. Not canon.

## 5.1 Seeing Is Not Deciding

Part 4 made the field visible.

Action changes future-space. Harm can be a narrowing of options, reachability, or usable information. Benefit can be an expansion. Time can make correction real or ceremonial. Opportunity can expire. Residue can remain.

That description matters.

But seeing the field is not the same as deciding what may be done.

```trace
seeing_transition
  != deciding_transition
```

This distinction is easy to lose.

Once a system can describe harm, it may be tempted to treat description as control. Once an institution can map a burden, it may be tempted to treat mapping as answerability. Once a model can classify a risk, it may be tempted to treat classification as permission. Once a person can explain a dirty conflict, they may be tempted to treat explanation as moral cleansing.

Mechanical Ethics refuses that move.

A description can make action harder to deny. It can make responsibility harder to evade. It can show that a route is too slow, a burden is hidden, a correction is late, or a scope has been erased.

But a description does not itself justify what happens next.

This is not a technical point. It is a moral safety rail.

If seeing became deciding, then whoever controlled the description would control the permission. The mapper would become the judge. The analyst would become the authority. The system would produce its own absolution.

Mechanical Ethics begins prescription only after description, but it does not collapse into description.

## 5.2 Description Is Not Permission

The central firewall is simple:

```trace
description_is_not_permission :=
  seeing_a_transition
  != authorizing_that_transition
```

A case record may show that an action narrows one scope and expands another. That does not decide whether the trade is acceptable.

A clock may show that correction is likely to arrive before hardening. That does not decide whether the original burden was legitimate.

A residue ledger may show what remains. That does not decide whether the residue may be imposed.

A future-space comparison may show benefit. That does not decide whether the harmed scope can be used as the price of that benefit.

The same rule applies in reverse. A description of harm does not automatically settle that an action must stop. Some action may be necessary. Inaction may be worse. Every available route may carry loss. A dirty conflict may require a constrained choice rather than a clean refusal.

So description is neither permission nor prohibition.

It is the field that prescription must answer.

```trace
TRACE_description_or_case_description := field_seen
ME_prescription := answer_to_field_seen
```

That answer needs its own route. It must say which structural pressure creates which duty or constraint. It must name affected scopes. It must identify uncertainty. It must record timing. It must acknowledge residue. It must say what remains unresolved.

Without that route, prescription is only assertion.

## 5.2.1 Benefit Is Not Permission

The same firewall applies to claimed good.

A claimed benefit may be real. It may open opportunity, restore reachability, improve information, build trust, coordinate action, or protect a vulnerable scope. Mechanical Ethics must be able to say that. It is not a harm-only grammar.

But benefit is not permission.

```trace
benefit_is_not_permission :=
  seeing_or_claiming_expansion
  != authorizing_the_transition
```

A benefit claim must still answer for the field it changes.

```trace
benefit_audit :=
  what_expands
  + for_whom
  + what_contracts
  + who_carries_burden
  + what_uncertainty_remains
  + what_correction_stays_live
  + what_residue_may_remain
```

This blocks the mirror-image misuse of Mechanical Ethics. The book must not only stop actors from using visible harm as a managed cost. It must also stop actors from using visible benefit as absolution.

An expansion for one scope can be candidate-good and still leave another scope narrowed. A safety improvement can be real and still depend on coercion. A repair can reopen one path and still leave trust damaged. A system can create opportunity and still extract burden from those least able to contest.

So ME asks a harder question: what becoming is supported, what becoming is blocked, who carries the cost, and what remains answerable?

## 5.3 Findings Are Not Authorizations

Mechanical Ethics uses findings.

A finding may say that no structural objection was found in the available record. It may say proceed only with named constraints. It may say pause, halt, escalate, repair, compensate, hold, or return an unresolved value-crux.

But a finding is not authorization.

```trace
ME_output :=
  finding
  + recommendation
  + duty
  + constraint

ME_output != permission_to_act
```

This is especially important for the weakest-sounding finding:

```trace
NO_STRUCTURAL_OBJECTION_FOUND
```

That phrase must not be inflated.

It does not mean the action is right. It does not mean the action is legal. It does not mean the action is wise, safe, legitimate, kind, complete, or free of future challenge.

It means only this: within the available record, for the named decision or question, Mechanical Ethics did not identify a structural objection under its current rules.

That finding can be wrong if the record is wrong. It can become stale if facts change. It can be too narrow if the question was framed too narrowly. It can be unsafe if the actor treats it as moral clearance.

The actor remains responsible.

The finding does not replace judgment. It disciplines judgment.

## 5.4 The Actor Still Answers

A person cannot hand their responsibility to a protocol and be done.

An institution cannot say, "the process allowed it," if the process hid the loss, erased the scope, or arrived too late.

A system designer cannot say, "the model recommended it," if the model's estimate was contaminated, the route was opaque, or the harmed scope could not contest.

A state cannot say, "the emergency required it," if the emergency is being used to launder prior failure.

Mechanical Ethics keeps asking who still answers after the finding.

```trace
responsibility_remaining :=
  actor
  + role
  + control
  + decision_time_evidence
  + capacity_to_act
  + residue_after_finding
```

This matters because many modern systems are built to dissolve answerability. Responsibility is spread across departments, software, policies, contracts, models, queues, committees, vendors, and dashboards. Each part says it only followed its local rule. The whole path harms, but no part speaks as the actor.

Mechanical Ethics does not accept that disappearance.

Distributed action can create distributed responsibility. Delegation can move tasks, but not erase answerability. Automation can change the mechanism, but not remove the duty to understand, monitor, constrain, and repair the mechanism.

The actor still answers because the world still changes.

## 5.5 When Description Creates Pressure

Description is not permission, but it is not neutral either.

Once a harm path is visible enough, continuing without answer becomes harder to defend. Once a delay is visible enough, letting it harden may become part of the mechanism. Once a correction channel is known to fail, using it as proof of fairness becomes laundering. Once a scope is known to be affected, erasing it becomes an active distortion.

```trace
visible_path_pressure :=
  path_seen
  + affected_scope_seen
  + timing_seen
  + available_response
  -> duty_to_answer
```

This is not hindsight blame. The pressure depends on decision-time evidence, not perfect knowledge after the fact.

But a live warning matters. A contested estimate matters. A known failure mode matters. A repeated pattern matters. A reachable correction path matters. A practical ability to slow or pause matters.

The book will later build responsibility more carefully. For now, the point is narrower: visibility raises the cost of pretending not to know.

If the actor sees enough to know that another scope may be narrowed, that uncertainty is material, and that time may close the path before repair arrives, then the actor cannot treat the action as ordinary until those pressures are answered.

The same is true when the actor sees enough to know that a claimed good may depend on another scope carrying hidden burden. Positive value does not remove the duty to answer. It increases the need to keep the account honest.

This is the bridge to the first gates.

## 5.6 The Human Firewall

This part replaces technical interface machinery in the human spine.

The operator schema belongs in case records and appendices. The reader does not need to know every field before understanding the principle.

The principle is this:

```trace
human_firewall :=
  seeing_is_not_deciding
  + description_is_not_permission
  + benefit_is_not_permission
  + findings_are_not_authorizations
  + actor_still_answers
```

The book can now move from description to constraint.

It has not solved morality. It has not ranked all values. It has not decided personhood, coercion, desert, dignity, or acceptable sacrifice.

It has done something smaller and necessary.

It has blocked the easiest misuse of the whole project: using a better description of harm as a better excuse to continue harming, or using a better description of benefit as a better excuse to ignore cost.

The next part states the first gates.

They are not the whole of Mechanical Ethics. They are the minimum constraints that follow once action under uncertainty, interaction, future-space, time, and description-not-permission have been accepted.
