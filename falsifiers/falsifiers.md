# Falsifiers and Failure Conditions

This repository should not protect itself from failure.

The Correction Window Test is useful only if it helps people see something real about safeguards, timing, correction, and hardening.

If it does not, it should be revised, narrowed, or abandoned.

## Falsifiers for the Correction Window Test

### 1. It adds no value beyond ordinary review

If ordinary administrative-justice, safety, legal, or audit checklists identify the same problems with equal or better clarity, the test has no distinctive public value.

Result: demote to a plain checklist or citation aid.

### 2. It overreads missing evidence

If users repeatedly turn Unknown into Failure or Safety, the test is encouraging false inference.

Result: strengthen Unknown discipline or narrow the test.

### 3. It treats every problem as timing

If the test repeatedly ignores distribution, rights, discrimination, legitimacy, coercion, poverty, power, or material deprivation because it is looking for clocks, it is over-compressing.

Result: state that timing is only one diagnostic layer.

### 4. It cannot distinguish live protection from after-the-fact review

If users cannot reliably tell whether a safeguard acts before or after hardening, the classification scheme is too weak.

Result: revise categories or abandon classification.

### 5. It cannot be used by strangers

If a person outside the project cannot apply the worksheet without private context, the public surface has failed.

Result: simplify the worksheet and examples.

### 6. It hides judgement behind symbols

If notation or internal language makes the tool harder to check, remove it from the public surface.

Result: plain English controls.

### 7. It becomes a validation claim

If the test is used to claim that Mechanical Ethics is proven, AI alignment is solved, or a system is fully ethical, the public claim has exceeded the evidence.

Result: correct or withdraw the claim.

## Falsifiers for public-record series

### 1. No trend

If the long series does not show pressure, decline, delay, or outbidding, the pressure claim weakens.

### 2. Recovery under pressure

If the series shows stress followed by timely recovery, the honest result may be resilience, not collapse.

### 3. Case mix explains the movement

If changes in demand composition explain the latency or backlog pattern, the correction-pressure interpretation weakens.

### 4. Measurement break explains the movement

If a methodology change explains the apparent worsening or recovery, the series cannot support the stronger claim.

### 5. Capacity denominator is missing

If there is no denominator such as staff, sitting days, budget, or available hours, pressure interpretation remains limited.

### 6. Segment clock is mistaken for end-to-end clock

If a complaint-stage series is presented as the affected person's full repair path, the public claim fails.

### 7. Basket is not additive

If spending categories overlap or are not classificationally compatible, they must not be added into a total.

## What success would look like

A useful public surface lets a stranger:

- inspect the source;
- understand the claim;
- see the caveats;
- reproduce the table or example;
- extend the series;
- challenge the interpretation;
- identify what would change the conclusion.

## What failure would look like

A failed public surface requires the reader to accept the whole theory before they can inspect the claim.

A failed public surface turns examples into proof.

A failed public surface hides uncertainty.

A failed public surface produces only agreement from sympathetic readers.

## Standing rule

A result against the preferred thesis must be preserved with the same prominence as a supporting result.

A method that cannot return a clear "no" is not yet a method.
